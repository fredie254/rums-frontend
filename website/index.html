Enterprise Security

Built Secure from
the Ground Up

Security is not a feature layer — it is baked into every API endpoint, every database column, and every HTTP response header.

AES-256-GCM Field Encryption

PII fields (name, phone, email, ID number) are encrypted at rest with a per-record nonce. The enc1: prefix format allows transparent field migration.

TOTP Multi-Factor Authentication

RFC 6238 TOTP with HMAC-SHA1, 30-second period, 6-digit codes. Compatible with Google Authenticator and Authy. 10-minute MFA challenge TTL with automatic cleanup.

CSRF & Security Headers

Synchronised token CSRF pattern with hash_equals(). Full security header suite: HSTS, CSP, X-Frame-Options, Referrer-Policy, Permissions-Policy.

Rate Limiting

120 requests per minute per IP/token enforced at the API middleware layer. Sliding window stored in api_rate_limits table with Retry-After response headers.

SQL Injection Prevention

100% PDO prepared statements with bound parameters throughout. No dynamic SQL construction. Input validation at every API boundary before any database interaction.

Immutable Audit Logs

Append-only audit_logs table with no UPDATE or DELETE grants for the application user. Every data change recorded with before/after JSON snapshots.

Common Questions

Frequently Asked
Questions

Does RUMS work with Safaricom M-Pesa in Kenya?
Yes — RUMS is natively integrated with Safaricom's Daraja API using the C2B STK Push flow. When a tenant clicks "Pay Now", an M-Pesa payment prompt is sent directly to their phone. Once they enter their PIN, Safaricom calls your secure callback endpoint and RUMS automatically marks the invoice as paid. The system supports both Sandbox (for testing) and Production environments.
What happens if M-Pesa STK Push fails?
If the STK Push times out or is cancelled, RUMS captures the failure reason from the Daraja callback (ResultCode), logs it in the mpesa_transactions table, and notifies the tenant by SMS. The tenant can retry payment immediately. Common failure codes (1032 = cancelled, 1037 = timeout) are handled gracefully with specific user messages.
Is tenant personal data kept secure and private?
Absolutely. All PII (names, phone numbers, email addresses, ID numbers) is encrypted at rest using AES-256-GCM with a per-record nonce. Passwords use bcrypt with cost factor 12. RUMS also implements full GDPR compliance — tenants can download all their data (Art. 20) or request permanent deletion/anonymisation (Art. 17) directly from their portal.
How does the automated invoice generation work?
A cron job runs at 00:05 on the 1st of every month. It queries all active leases, calculates rent + service charges + utilities for each tenant, and inserts invoices into the invoices table. Tenants receive an SMS and email notification the moment their invoice is generated. Partial payments, outstanding balances, and prorated amounts for mid-month lease starts are all handled automatically.
Can multiple landlords share one RUMS installation?
Yes. RUMS is multi-landlord by design. Each landlord is assigned a user account and can only see properties they own. Admins and Managers have cross-portfolio visibility. Row-level security in the API middleware ensures that landlord-scoped API tokens only return data for their assigned properties, even on shared endpoints.
What server requirements does RUMS need?
RUMS runs on any standard Linux server with PHP 8.1+, MySQL 8.0+, and Apache or Nginx. Minimum specs for a single-server deployment: 2 vCPU, 4 GB RAM, 20 GB SSD. For the M-Pesa callback endpoint, a public HTTPS domain is required (Let's Encrypt works perfectly). There is no cloud provider lock-in — deploy on-premises, on a VPS, or on any cloud.
Does RUMS support Multi-Factor Authentication for all users?
Yes. RUMS implements TOTP MFA compliant with RFC 6238 (the same standard used by Google Authenticator and Authy). Users can enrol via a QR code in Account Settings, with 8 one-time backup codes generated at setup. MFA can be made mandatory for admin and manager roles. The 10-minute MFA challenge token uses a secure pending_token table with automatic cleanup via cron.
